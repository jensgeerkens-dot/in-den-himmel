## KRITISCH

### K1 — Kontrast: Captions/Akzentfarbe über dem hellen Himmel-Gradient unlesbar
Die Szene wechselt den Hintergrund von Weltraum-Dunkelblau bis Himmel-Hellblau (`#87ceeb`/`#5aa0d8`). Text wird aber durchgehend in `--accent #6fc4ff` bzw. halbtransparentem Weiß gezeichnet, abgesichert nur durch `text-shadow`. Über dem hellblauen Bereich (untere Atmosphäre, wo die meisten Tiere/Flugzeuge sitzen) ergibt das:

| Element | Vordergrund | Hintergrund (hellblau) | Ratio | WCAG |
|---|---|---|---|---|
| `.sky-item .alt` (Höhenangabe) | `#6fc4ff` | `~rgb(90,160,216)` | **1,48:1** | ✗ (<4,5) |
| `.year` (Jahreszahl) | `#6fc4ff` | hellblau | **1,48:1** | ✗ |
| `.atmo-label` | `rgba(255,255,255,.42)` | hellblau | **1,6:1** | ✗ |
| `.note-badge`-Text | `rgba(255,255,255,.78)` | hellblau | **2,3:1** | ✗ |
| `.sky-item .name` (Weiß) | `#fff` | hellblau | **2,5:1** | ✗ |

`text-shadow` zählt für WCAG nicht als zuverlässiger Kontrast. Im dunklen oberen Bereich sind dieselben Farben unkritisch (`#6fc4ff` auf `#060c20` = 10,2:1), das Problem ist die **untere Hälfte der Seite**.

**Fix:** Captions auf eine eigene halbtransparente, dunkle Plakette setzen, statt sich auf den variablen Hintergrund zu verlassen:
```css
.sky-item figcaption{
  display: inline-block;
  padding: 4px 10px;
  border-radius: 8px;
  background: rgba(8,14,30,0.62);   /* fester dunkler Träger unter dem Text */
}
.sky-item figcaption .name{ color: #fff; }            /* 16:1 auf der Plakette */
.sky-item figcaption .alt{ color: #a9dbff; }          /* helleres Blau, ~7:1 auf dunkel */
```
Alternativ den Akzent generell aufhellen (z. B. `--accent: #a9dbff;`) — löst aber die Hellblau-Fälle der weißen Captions nicht. Die Plakette ist robuster, weil sie vom Szenen-Hintergrund unabhängig wird.

### K2 — `prefers-reduced-motion` greift nicht für die Haupt-Bewegung (Teleport-Scroll & Init-Sprung)
Die störendste Bewegung ist das lange Smooth-Scrollen. `script.js` respektiert die Präferenz beim Teleport zwar via `prefersReduced()` (Z. 533), **aber** `html.ready{ scroll-behavior: smooth }` (CSS Z. 16) gilt für *jeden* programmatischen `scrollTo`. Die Media-Query in CSS (Z. 493–498) setzt `scroll-behavior: auto` zwar zurück — gut —, jedoch:
- Die Parallax-Wolken (`updateParallax`, Z. 335) bewegen sich **immer**, unabhängig von der Präferenz. Bei Reduced-Motion sollte der Parallax-Versatz entfallen.
- Das Hover-Popup wird zwar in CSS (`#hover-popup{ animation: none }`) ruhiggestellt, taucht aber weiterhin schlagartig auf — akzeptabel, aber Parallax ist eine echte, kontinuierliche Bewegung, die WCAG 2.3.3 (Animation from Interactions) / Motion-Empfindlichkeit betrifft.

**Fix (JS):** Parallax bei Reduced-Motion deaktivieren:
```js
function updateParallax(st){
  if (prefersReduced()) return;            // Wolken statisch lassen
  for (const el of cloudEls){
    const f = parseFloat(el.dataset.factor);
    el.style.transform = `translateY(${st * f}px)`;
  }
}
```
(Beim Wechsel der Präferenz ggf. einmal `transform` zurücksetzen.)

### K3 — 155 fokussierbare Items ohne Sprungmöglichkeit: Tastatur-Falle „durch die ganze Szene tabben"
Jedes `.sky-item` bekommt `tabIndex = 0` (Z. 144). Damit muss ein Tastaturnutzer durch **bis zu 155 Elemente** tabben, um z. B. von den Controls zum Modal oder ans Seitenende zu gelangen. Es gibt keine Skip-Links und keine Landmark-Navigation. Das ist eine schwere Bedienbarkeitsbarriere (WCAG 2.4.1 Bypass Blocks).

Verschärfend: Die DOM-Reihenfolge der Items entspricht der **Sortierung nach y** (Z. 122–124, von oben/hoch nach unten/niedrig), während die Seite **von unten nach oben** gelesen/gescrollt wird. Die Tab-Reihenfolge läuft also entgegen der visuellen Leserichtung (WCAG 2.4.3 Focus Order).

**Fix:**
1. Skip-Link am Anfang des `<body>` und das Teleport-Menü als echte Sprungnavigation (siehe H3) — gibt Tastaturnutzern eine Abkürzung.
2. DOM-Reihenfolge an die Leserichtung anpassen: Items aufsteigend nach Höhe einfügen (`b.y - a.y` statt `a.y - b.y` für die DOM-Append-Reihenfolge), damit Tab = „aufsteigen" wie das Scrollen. Die Lane-Berechnung kann auf der nach-y-sortierten Liste bleiben; nur die Append-Reihenfolge umkehren.
3. Optional: Items, die nicht im Viewport sind, müssen ohnehin nicht alle erreichbar sein — aber `content-visibility:auto` entfernt sie *nicht* aus der Tab-Ordnung. Eine echte Lösung ist die Sprungnavigation.

## HOCH

### H1 — `<details>`-Teleportliste ist nicht tastaturbedienbar
Die `<li>`-Einträge im Teleport-Menü reagieren nur auf `click` (Z. 529) und sind keine fokussierbaren/aktivierbaren Elemente — kein `tabindex`, keine `role`, kein Keydown-Handler. Mit der Tastatur lässt sich das Menü via `<summary>` öffnen, aber **kein Ziel auswählen**. (WCAG 2.1.1 Keyboard)

**Fix:** Echte Buttons statt klickbarer `<li>`:
```js
for (const t of TELEPORT_TARGETS){
  const li = document.createElement("li");
  const btn = document.createElement("button");
  btn.type = "button";
  btn.className = "teleport-btn";
  btn.textContent = t.label[lang];
  btn.addEventListener("click", () => { /* … scrollTo … */ });
  li.appendChild(btn);
  frag.appendChild(li);
}
```
CSS für `.teleport-btn` analog zum bisherigen `li`-Styling (volle Breite, transparent, `:hover`/`:focus-visible`-Stil).

### H2 — Sprachschalter aktualisiert `aria-label`s/Inhalte nicht vollständig & meldet aktiven Zustand nicht
- Die `aria-label`-Texte der UI sind **hartcodiert deutsch** im HTML: `role="group" aria-label="Sprache"` (Z. 39), `aria-label="Schließen"` (Modal-Close, Z. 60). Bei EN bleiben sie deutsch. `applyStaticI18n` (Z. 98) übersetzt nur `textContent` von `[data-i18n]`, nicht diese Labels.
- Die Lang-Buttons zeigen den aktiven Zustand nur per CSS-Klasse, nicht semantisch: kein `aria-pressed` bzw. `role="radio"`+`aria-checked`. Screenreader sagen nicht, welche Sprache aktiv ist.

**Fix:**
```html
<div id="lang-toggle" role="group" data-i18n-aria="langGroup" aria-label="Sprache">
  <button data-lang="de" aria-pressed="true">DE</button>
  <button data-lang="en" aria-pressed="false">EN</button>
</div>
```
```js
// in setLang():
document.querySelectorAll("#lang-toggle button").forEach((b) => {
  const on = b.dataset.lang === lang;
  b.classList.toggle("active", on);
  b.setAttribute("aria-pressed", String(on));
});
// I18N um langGroup/close/jump-Labels ergänzen und in applyStaticI18n() auch
// data-i18n-aria → setAttribute("aria-label", …) verarbeiten.
modal.querySelector(".modal-close")
     .setAttribute("aria-label", lang === "de" ? "Schließen" : "Close");
```

### H3 — Teleport-Liste sollte als `role="menu"`/Navigation und tastaturzyklisch fungieren
Auch nach H1-Fix bleibt die Liste eine lose Button-Sammlung. Für eine echte Sprungnavigation (die K3 entschärft) sollte sie eine Landmark/Überschrift haben und nach Auswahl den Fokus sinnvoll setzen (z. B. auf das gesprungene Ziel oder den Höhenmesser), sonst landet der Fokus nach Schließen des `<details>` im Nichts.

**Fix:** Nach `scrollTo` Fokus auf ein Ziel setzen, das der neuen Position entspricht, z. B. das oberste sichtbare `.sky-item` mit `el.focus({preventScroll:true})`, oder zumindest den Fokus zurück auf das `<summary>` legen, damit er nicht verloren geht.

### H4 — Höhenmesser als Live-Region fehlt; `aria-hidden="true"` macht ihn für SR komplett stumm
`#altimeter` ist `aria-hidden="true"` (Z. 31). Das ist für die *Dekoration* (Balken) richtig, aber die aktuelle Höhe + Atmosphärenschicht ist die **Kerninformation** der Seite und für Screenreader-Nutzer dadurch gar nicht zugänglich — sie scrollen „blind", ohne zu erfahren, auf welcher Höhe sie sind.

**Fix:** Höhenmesser nicht komplett verstecken. Den numerischen Teil als dezente Live-Region zugänglich machen (gedrosselt, sonst Dauerfeuer):
```html
<aside id="altimeter">
  <div class="alt-value" aria-live="polite" aria-atomic="true">
    <span id="alt-number">0</span>
  </div>
  <div class="alt-layer" id="alt-layer-name">Troposphäre</div>
  <div class="alt-bar" aria-hidden="true"><div class="alt-bar-fill" id="alt-bar-fill"></div></div>
</aside>
```
Im rAF-Loop nur bei tatsächlicher Textänderung schreiben und idealerweise per Throttle (z. B. nur alle ~600 ms) aktualisieren, damit die Live-Region nicht jeden Frame spricht.

### H5 — `alt`-Texte sind nur der Name; Emoji-Platzhalter werden vorgelesen
- `<img alt="${it.name[lang]}">` (Z. 153): Der Alternativtext ist nur der Name. Da Name **und** Höhe ohnehin als Text in der `figcaption` stehen, ist das Bild im Grunde redundant → besser `alt=""` (dekorativ), damit der SR nicht „Haussperling Grafik, Haussperling, 50 m" doppelt liest. Aussagekräftiger Alt-Text wäre nur sinnvoll, wenn das Bild Information trägt, die die Caption nicht hat.
- Der Emoji-Platzhalter `<div class="placeholder">🐦</div>` (Z. 155) hat **kein** `aria-hidden`. Fällt das Bild aus, liest der SR das Emoji als „Vogel" o. Ä. vor.

**Fix:**
```js
<img src="${it.img}" alt="" loading="lazy" decoding="async" ...>
<div class="placeholder" aria-hidden="true" style="display:none">${it.emoji}</div>
```
Da `aria-label` der `figure` (Z. 146) Name + Höhe bereits trägt und `role="button"` gesetzt ist, sollten die inneren Texte für den SR-Namen nicht doppelt zählen — siehe M1.

### H6 — `role="button"` auf `<figure>` verschluckt den Caption-Inhalt; kein klarer Aktivierungs-Hinweis
Die `<figure>` hat `role="button"` + `aria-label="Name – Höhe"` (Z. 145–146). `role="button"` macht das Element zu einem Blattknoten: Der Accessible Name kommt aus dem `aria-label`, die innere `figcaption`/`<img>` wird i. d. R. nicht zusätzlich vorgelesen — das ist hier eigentlich erwünscht. Aber:
- Beim Aktivieren öffnet sich ein Modal; das sollte angekündigt werden via `aria-haspopup="dialog"`.
- Das Hover-Popup zeigt sehend den Kurzfakt; Tastatur-/SR-Nutzer bekommen den Fakt **erst im Modal** — vertretbar, aber `aria-label` könnte um einen Hinweis ergänzt werden.

**Fix:**
```js
el.setAttribute("role", "button");
el.setAttribute("aria-haspopup", "dialog");
el.setAttribute("aria-label", `${it.name[lang]}, ${formatAlt(it.altitude_m)}`);
```

## MITTEL

### M1 — `#hover-popup` korrekt `aria-hidden`, aber `focusin` zeigt es trotzdem für Tastatur
`#hover-popup` ist `aria-hidden="true"` (HTML Z. 55) — richtig, da der Inhalt (Name/Höhe/Fakt) redundant bzw. im Modal verfügbar ist. Das Popup wird auch bei `focusin` eines Items eingeblendet (Z. 512–515), ist aber `pointer-events:none` und `aria-hidden` → rein visuell. Das ist akzeptabel. **Aber:** Der Fakt im Popup ist die einzige Stelle, an der die Kurzinfo *ohne* Modal-Öffnen erscheint; SR-Nutzer verpassen sie. Kein Blocker, da das Modal die volle Info liefert.

**Fix (optional):** Den Fakt zusätzlich in die `figure` als visuell verstecktes `aria-describedby`-Ziel legen, falls die Kurzinfo ohne Modal verfügbar sein soll.

### M2 — Szene bei offenem Modal nur via `aria-hidden`, Hintergrund bleibt scrollbar & fokussierbar
`openModal` setzt `sceneEl.setAttribute("aria-hidden","true")` (Z. 416) — gut für SR. Aber:
- Die `#controls` (Sprachschalter, Teleport) und `#altimeter` liegen **außerhalb** von `#scene` und werden **nicht** `aria-hidden`. SR-Nutzer können aus dem Modal heraus zu ihnen navigieren (virtueller Cursor), obwohl die Fokusfalle nur `Tab` abfängt.
- Der Body bleibt scrollbar (kein `overflow:hidden`), der Hintergrund scrollt unter dem Modal weg.
- `aria-hidden` auf einem Element, das den Fokus enthielt, kann problematisch sein — hier unkritisch, da Fokus ins Modal wandert.

**Fix:** Beim Öffnen `document.body.style.overflow="hidden"` und auch `#controls`/`#altimeter` per `inert` (oder `aria-hidden`) ausblenden; beim Schließen zurücksetzen. `inert` ist robuster als `aria-hidden`, weil es Fokus *und* SR blockt:
```js
function openModal(it){ /* … */
  document.body.style.overflow = "hidden";
  document.getElementById("controls").inert = true;
  document.getElementById("altimeter").inert = true;
}
function closeModal(){ /* … */
  document.body.style.overflow = "";
  document.getElementById("controls").inert = false;
  document.getElementById("altimeter").inert = false;
}
```

### M3 — Fokusfalle ignoriert Elemente mit positivem/dynamischem Zustand & sichtbarkeit
`trapFocus` (Z. 431–439) sammelt Fokussierbare statisch bei jedem Tab. Funktioniert hier (Close-Button + ggf. Quell-Link). Schwächen:
- Der einzige garantiert fokussierbare Knoten ist oft nur `.modal-close`; gibt es keinen Quell-Link, zykliert Tab auf sich selbst — ok.
- Bei `display:none`/versteckten Fokussierbaren würde `querySelectorAll` sie fälschlich einschließen. Aktuell nicht der Fall, aber fragil.

**Fix:** Sichtbarkeit prüfen (`offsetParent !== null` bzw. `getClientRects().length`) beim Filtern der Fokussierbaren. Niedrige Priorität, da aktuell stabil.

### M4 — Kein sichtbarer Fokus auf vielen interaktiven Elementen
`:focus-visible` ist nur für `.sky-item` definiert (CSS Z. 161). Es fehlt für: Lang-Buttons (`#lang-toggle button`), `<summary>`, Teleport-Einträge, `.modal-close`, `#start-btn`, Quell-Link im Modal. Browser-Default-Outline greift teils, wird aber durch dunkle UI/`border:none` schlecht sichtbar (WCAG 2.4.7).

**Fix:** Globale Fokus-Regel:
```css
#lang-toggle button:focus-visible,
#teleport-menu summary:focus-visible,
.teleport-btn:focus-visible,
.modal-close:focus-visible,
#start-btn:focus-visible,
.modal-meta a:focus-visible{
  outline: 2px solid var(--accent);
  outline-offset: 2px;
  border-radius: 6px;
}
```

### M5 — Keine Überschriftenstruktur / Landmarks für Orientierung
- `<h1>` steckt nur im Start-Overlay (Z. 73); ist es weggeblendet (`display`? nein — nur `opacity:0`+`pointer-events:none`), bleibt es für SR lesbar, liegt aber außerhalb des Hauptinhalts. Es gibt **keine** `<h2>` in der eigentlichen Szene; die Akt-Banner (`.section-title`) und Atmosphären-Labels sind reine `<span>`s ohne Heading-Semantik.
- Landmarks: `<main id="scene">` und `<nav id="controls">` existieren — gut. Aber die 155 Items liegen flach in `#objects-layer` ohne gruppierende Struktur/Überschriften, sodass keine Sprungmarken zwischen den „Akten" existieren.

**Fix:** Akt-Banner als Überschriften auszeichnen, damit SR-Nutzer per Heading-Navigation springen können:
```js
el.innerHTML = `<h2 class="section-title">${s.name[lang]}</h2>`;  // statt <span>
```
(Optisch identisch via CSS.) Das gibt eine grobe Gliederung „Atmosphäre / Weltraum / Sonnensystem / …" für die Heading-Liste des SR.

### M6 — `lang`-Attribut auf Items wechselt korrekt, aber gemischtsprachige Inhalte ohne `lang`-Markierung
`document.documentElement.lang = lang` wird in `applyStaticI18n` gesetzt (Z. 99) — gut, der Sprachwechsel aktualisiert `<html lang>`. **Aber** wissenschaftliche Namen (`it.sci`, z. B. „Passer domesticus") und Eigennamen bleiben latein/englisch; im DE-Modus liest ein deutscher SR sie deutsch aus. Niedrig-mittel, betrifft Aussprache.

**Fix (optional):** `<p class="modal-sci" lang="la">` für lateinische Taxa, bzw. die wenigen englischen Eigennamen mit `lang="en"` auszeichnen.

## NIEDRIG

### N1 — Modal-Meta-Kontrast grenzwertig
`.modal-meta` `rgba(255,255,255,0.5)` auf Modal-Hintergrund `#14213f` ≈ **4,98:1** — knapp über 4,5, aber bei `font-size:0.8rem` (kein Großtext) ohne Reserve. `.modal-sci` (0.6) ≈ 6,5:1 ist ok. Empfehlung: Meta auf `rgba(255,255,255,0.62)` anheben (→ ~6:1).

### N2 — Start-Overlay-Button: `#start-btn` Fokus nach Schließen
Nach `hideOverlay` (Z. 594) wird der Fokus nicht aktiv gesetzt; er bleibt auf dem nun `pointer-events:none`/`opacity:0`, aber noch im DOM vorhandenen `#start-btn`. Tastaturnutzer „verlieren" den Fokuskontext. Empfehlung: nach dem Ausblenden Fokus auf ein sinnvolles Element legen (z. B. erster Lang-Button oder `#scene`-Skip-Target) und das Overlay nach der Transition `inert`/`hidden` setzen.

### N3 — Klick-zum-Schließen auf Backdrop ohne Tastatur-Äquivalent (vorhanden via Esc — ok)
`modal.addEventListener("click", … target===modal … closeModal)` (Z. 442) — Backdrop-Klick schließt. Esc schließt ebenfalls (Z. 444). Tastaturseitig also abgedeckt. Kein Handlungsbedarf, hier nur zur Vollständigkeit dokumentiert.

### N4 — `scroll-hint`/Bob-Animation: Reduced-Motion abgedeckt
`#scroll-hint{ animation: bob }` wird unter `prefers-reduced-motion` korrekt auf `none` gesetzt (CSS Z. 495). Popup-Animation ebenfalls (Z. 497). Korrekt — kein Fix. Lücke nur beim Parallax → siehe K2.
